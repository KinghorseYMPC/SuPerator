# Log Compliance Strategy

## Official Requirement Summary

Each submitted task must include `task{N}_logs.log`. Every line must be one valid JSON object. Each object must contain a timezone-aware ISO 8601 `timestamp`, non-negative `elapsed_seconds`, and the Agent LLM output in `response` or `tool_calls`. The first and last timestamps in one task log must be no more than 12 hours apart. The submitted `code/` directory must be explainable from the recorded LLM call history.

## Official Proxy Tool

The official sample materials include `task_log_sample/openai-log/proxy.py`. The proxy forwards OpenAI-compatible API requests and writes JSONL records for completed LLM calls. The documented launch pattern is:

```bash
python proxy.py --port 8080 --target https://api.openai.com --log-dir ./logs
```

After an Agent API base URL is pointed at `http://localhost:8080`, the proxy can capture LLM responses into a log directory.

## Log Types In This Project

`development_summary_log`: generated by `src/agent/task_log_writer.py`. It is used for local engineering validation and submission pipeline checks. It is structurally aligned with JSONL requirements, but it is a curated development summary and is not a complete API response capture.

`api_proxy_llm_log`: generated by the official proxy from real LLM API traffic. This is closer to the final competition provenance requirement because it records actual LLM call responses.

## Current Risks

Submitting only `development_summary_log` may not prove that `code/` corresponds to the full LLM call history. If Codex UI sessions cannot export complete LLM response JSONL, final compliance remains a provenance risk even when local structural validation passes.

Do not forge LLM call logs. Do not claim that development summary logs are equivalent to official proxy captures.

## Strategy

Before A4, continue using `development_summary_log` for local engineering loops, validator tests, and submission packaging checks.

Before final official submission, prefer an API-based Agent run routed through `task_log_sample/openai-log/proxy.py` so the final artifact can use `api_proxy_llm_log`. If the platform provides a complete export of Codex LLM calls, that export may also be evaluated as the provenance source.

## Validator Strategy

The structural validator checks JSONL parsing, object shape, required fields, timezone-aware timestamps, non-negative elapsed time, response or tool call content, placeholder text, and 12-hour span. Provenance cannot be fully proven in code, so the validator reports a warning when it recognizes `development_summary_log`.

## Code Correspondence Strategy

Final logs should include code generation, code modification, tests, submission artifact generation, failures, results, and conclusions. Experiment registry entries and git commits can help map logs to code state, but the final submission should still include a traceable Agent operation history.
